# Task 1

The current code works well as a proof of concept because it provides a complete end-to-end training pipeline, from loading sequence data through training, evaluation, and saving outputs. The training loss decreases with a larger sample size (n=3000), which shows that the model is learning from the data rather than failing completely. Normalizing the fitness scores is also a good choice because it makes regression training more stable. In addition, the model is already set up to handle padded sequences up to a larger maximum length, which gives some flexibility for future datasets with longer proteins.

The main issues are related to experimental rigor and scalability rather than basic functionality. The biggest problem is that the test set is being used as a validation set during training, so there is no clean held-out test set for final evaluation. Padding is also not handled correctly: the mask is not actually used, positional encodings are added to padded tokens, and mean pooling includes padding positions, which would become a more serious problem when scaling to datasets with sequences of different lengths. More broadly, the current design is too tightly coupled, since data loading, preprocessing, training, evaluation, and plotting are all combined in one large class. That structure makes it difficult to compare multiple architectures and datasets or run systematic ablation studies. The model may also be somewhat large relative to the small synthetic dataset, since performance seems to improve only when the sample size is increased.

To better support the team’s goals, I would split `trainer.py` into separate modules for data, models, training, and experiments. Dataset-related operations such as loading, tokenization, normalization, padding, masking, and splitting should live in the data module, while model definitions should be placed behind a common interface so different transformer variants, positional encoding choices, and simpler baselines can be compared easily. The training loop, checkpointing, logging, and evaluation should live in a separate training module, and hyperparameters and experiment settings should move into config files so the team can run ablations and parameter sweeps without rewriting code. I would also add proper train, validation, and test splits, use seeded splits and seeded model initialization to make runs reproducible, and save run configurations, checkpoints, and metrics for each experiment so results can be traced and reproduced for publication.
